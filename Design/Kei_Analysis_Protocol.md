# Kei 解析协议设计

状态：当前方案已记录

目标：定义 Kei 如何将装备解析为可插入协议槽的解析协议 CD，并说明头部/身体装备与手部装备的不同处理方式。

## 一句话定位

解析协议是 Kei 将外部装备转化为可挂载 CD 后获得的扩展能力。

解析得到的 CD 不视为普通装备。协议槽也不视为普通装备栏。协议槽只负责放置 CD、读取 CD 中记录的装备数据，并驱动 Kei 的扩展系统。

当前解析协议分为两条实现路线：

```text
头部 / 身体装备解析 CD
-> 后台生成虚拟装备
-> 装备到 Kei 的隐藏虚拟装备栏
-> 进入 inventory.equipslots
-> 尽可能继承原装备组件、标签、回调和附加能力

手部装备解析 CD
-> 不生成虚拟装备
-> 不进入 inventory.equipslots
-> 只读取攻击力、位面伤害、移速倍率
-> 作为属性协议加到 Kei 自身面板
-> 多个手部解析 CD 可以叠加
```

## 协议分类

可插入协议槽的协议 CD 分为两类：

| 类型 | 来源 | 主要作用 | 挂载代价 |
| --- | --- | --- | --- |
| 战斗协议 | Boss 数据记录、战斗数据 CD | 提供 Boss 战斗数据、特殊战斗被动或主动能力 | 每个协议每 10 秒扣除 2 点电量 |
| 解析协议 | 解析手部、身体、头部装备 | 继承装备能力或提供装备属性加成 | 头/身虚拟装备每 10 秒扣除 2 点电量；手部属性协议每 10 秒扣除 2 点数据稳定性 |

手部装备解析 CD 在实现和结算上更接近战斗协议：它不作为装备存在，而是作为属性加成协议挂载到 Kei 身上。

## 可解析装备范围

第一版解析协议支持三类装备：

| 装备类型 | 原版装备槽 | 解析结果 | 生效方式 |
| --- | --- | --- | --- |
| 头部装备 | `EQUIPSLOTS.HEAD` | 生成头部解析 CD | 挂载后生成隐藏虚拟装备 |
| 身体装备 | `EQUIPSLOTS.BODY` | 生成身体解析 CD | 挂载后生成隐藏虚拟装备 |
| 手部装备 | `EQUIPSLOTS.HANDS` | 生成手部属性 CD | 挂载后作为属性加成协议生效 |

解析后得到的是一份无耐久 CD 数据。原装备本身不需要继续存在于协议系统中。

第一版解析入口：

- 装备解析工具临时使用 `wagstaff_tool_5` 贴图。
- 装备解析工具临时配方为 10 个金块。
- 当前先允许所有装备被解析。
- 解析不消耗原装备。
- 解析失败会消耗装备解析工具，但不给任何物品。
- 黑名单由后续继续添加，第一版预留代码入口。

CD 应至少记录：

```lua
{
    source_prefab = "source_equipment_prefab",
    source_slot = "head/body/hands",
    protocol_type = "analysis",
}
```

其中：

- `source_slot = "head"` 或 `"body"`：走隐藏虚拟装备路线。
- `source_slot = "hands"`：走属性协议路线。

## 头部与身体解析协议

头部和身体装备通常承载大量装备附加能力，例如护甲、防水、保温、隔热、电绝缘、抗性、特殊标签、装备时触发的被动效果等。

为了尽量兼容原版装备和强模组装备，头部与身体解析协议采用“虚拟装备”路线：

```text
解析原装备
-> 生成解析协议 CD
-> CD 插入协议槽
-> 后台生成一个来源于原装备 prefab 的隐藏虚拟装备
-> 清理虚拟装备的拾取、装备、归属等限制
-> 将虚拟装备装备到 Kei 的隐藏虚拟装备栏
-> 原装备组件、标签、回调和附加能力尽可能生效
```

玩家界面上只看到协议 CD，不看到虚拟装备。

## 隐藏虚拟装备栏

头部和身体解析协议需要隐藏虚拟装备栏承载虚拟装备。

建议为 Kei 建立与协议槽对应的隐藏虚拟装备栏：

```lua
EQUIPSLOTS.KEI_PROTOCOL_1 = "kei_protocol_1"
EQUIPSLOTS.KEI_PROTOCOL_2 = "kei_protocol_2"
EQUIPSLOTS.KEI_PROTOCOL_3 = "kei_protocol_3"
EQUIPSLOTS.KEI_PROTOCOL_4 = "kei_protocol_4"
EQUIPSLOTS.KEI_PROTOCOL_5 = "kei_protocol_5"
EQUIPSLOTS.KEI_PROTOCOL_6 = "kei_protocol_6"
EQUIPSLOTS.KEI_PROTOCOL_7 = "kei_protocol_7"
```

这些槽位不显示给玩家，也不作为普通装备栏操作。它们只用于让头部/身体虚拟装备进入 `inventory.equipslots`，从而参与原版装备遍历逻辑。

协议槽与隐藏虚拟装备栏一一对应。若某协议槽插入的是头部或身体解析 CD，则该槽驱动对应隐藏虚拟装备栏；若插入的是手部解析 CD，则该槽不生成虚拟装备，只提供属性加成。

## 虚拟装备生成规则

头部与身体虚拟装备由 CD 记录的 `source_prefab` 生成，但它不是原装备本体，而是解析协议的运行载体。

生成虚拟装备后，应进行协议化处理：

```lua
virtual.persists = false
virtual:AddTag("kei_virtual_equipment")

if virtual.components.equippable ~= nil then
    virtual.components.equippable.restrictedtag = nil
    virtual.components.equippable.equipslot = target_virtual_slot
end

if virtual.components.linkeditem ~= nil then
    virtual:RemoveComponent("linkeditem")
end

if virtual.components.inventoryitem ~= nil then
    virtual.components.inventoryitem.canbepickedup = false
    virtual.components.inventoryitem.cangoincontainer = false
end
```

必要时还应清理或忽略：

- 原装备的玩家装备限制。
- 原装备的归属绑定。
- 原装备的拾取限制。
- 原装备的容器放入限制。
- 原装备的耐久、燃料、腐烂等物品损耗。
- 原装备的容器能力。

如果来源装备带有容器，第一版可以选择不继承容器，或直接禁止解析带有容器的装备。实现时优先采用更稳定的方案。

解析协议 CD 自身只使用 Kei 的协议标签，例如：

```text
kei_protocol_cd
kei_analysis_protocol_cd
```

CD 不复制原装备的 `restrictedtag`、`linkeditem`、`canbepickedup` 等限制。

## 护甲与抗性结算

原版伤害结算由 `inventory:ApplyDamage()` 处理。

其基本逻辑为：

- 遍历 `inventory.equipslots` 中的所有装备。
- 若装备具有 `resistance` 且抵抗当前伤害，则直接抵抗伤害。
- 若装备具有 `armor`，则读取 `armor:GetAbsorption()`。
- 多件护甲同时存在时，吸收率取最高值。
- 护甲耐久损耗按吸收权重分摊。

头部和身体虚拟装备进入 `inventory.equipslots` 后，防御属性按照原版逻辑计算。也就是说，如果真实装备和多个虚拟装备都具有护甲吸收率，最终吸收率仍按原版最高值逻辑处理，而不是简单叠加。

这条规则用于保持原版防御系统的稳定性：

```text
头部 / 身体解析协议的防御能力
-> 按原版装备逻辑参与结算
-> 护甲吸收率取最高值
-> 不额外自定义叠加
```

## 无耐久处理

解析协议 CD 是无耐久数据，不应因为战斗、燃料、腐烂或使用次数损耗而损坏。

虚拟装备可以保留原装备的耐久、护甲、燃料、腐烂等组件用于参与原逻辑，但物品损耗不应真正破坏解析协议。

建议方案：

```text
保留原组件用于逻辑读取
拦截虚拟装备的耐久/燃料/腐烂损耗
虚拟装备不会因损耗移除
协议维持成本统一转化为 Kei 的电量消耗
```

如果某些装备能力强依赖自身损耗触发效果，可以作为后续个别适配对象处理。

## 手部解析协议

手部装备不生成虚拟装备，也不设定虚拟手部装备栏。

原因：

- 原版武器攻击通常只读取真实 `EQUIPSLOTS.HANDS`。
- 手部装备的核心通用价值主要集中在攻击力、位面伤害和移速倍率。
- 若将武器作为虚拟装备放入隐藏槽，普通攻击流程未必会读取它。
- 为了让手部解析协议稳定可控，第一版将其设计为属性协议。

手部解析 CD 的生效方式：

```text
手部装备解析 CD
-> 读取 source_prefab
-> 临时生成或查询原装备数据
-> 读取攻击力、位面伤害、移速倍率
-> 不保留虚拟装备实体
-> 将数值作为属性加成挂到 Kei 身上
```

### 第一版读取内容

| 数值 | 来源组件 | 处理方式 |
| --- | --- | --- |
| 攻击力 | `weapon.damage` 或 `weapon:GetDamage()` | 转化为 Kei 的攻击力加成 |
| 位面伤害 | `planardamage` 等位面伤害组件 | 转化为 Kei 的位面伤害加成 |
| 移速倍率 | `equippable.walkspeedmult` | 转化为 Kei 的移动速度修正 |

若原手部装备的 `walkspeedmult` 大于 1，则作为移速加成生效；若小于 1，则保留为减速代价，避免只继承收益而忽略负面效果。若装备没有 `walkspeedmult`，则不提供移速修正。

### 叠加规则

手部解析 CD 作为属性协议处理，可以多个同时叠加。

```text
多个手部解析 CD
-> 攻击力加成可以叠加
-> 位面伤害加成可以叠加
-> 移速倍率按后续平衡规则折算或叠加
```

建议第一版先采用直接叠加，后续根据强度再加入倍率、递减或上限：

```text
Kei 额外攻击力 = 所有手部解析 CD 攻击力加成之和
Kei 额外位面伤害 = 所有手部解析 CD 位面伤害加成之和
Kei 移速修正 = 所有手部解析 CD 移速修正汇总
```

手部解析 CD 不继承：

- 原武器耐久。
- 原武器主动技能。
- 原武器远程投射物逻辑。
- 原武器必须处于手持状态才能触发的逻辑。

攻击特效、视觉覆盖、动画覆盖等表现问题当前不作为解析协议设计边界单独讨论。

## 参考实现

### extra-equip-slot

`Reference/extra-equip-slot` 的核心思路是：

- 新增额外 `EQUIPSLOTS`，例如衣物栏、护符栏、背包栏。
- 用 `InventoryBar:AddEquipSlot()` 在 UI 上添加装备格。
- 将指定装备的 `equippable.equipslot` 改到新槽位。
- 因为这些装备仍然存在于 `inventory.equipslots` 中，原版 `inventory:ApplyDamage()` 会自然将额外装备一起纳入护甲、抗性和标签结算。

Kei 可以参考其“新增装备槽并让原版装备系统自然遍历”的思路，但 Kei 的虚拟装备栏不显示在 UI 上。

### fewer-slots

`Reference/fewer-slots` 证明了装备槽本质上可以通过 `equippable.equipslot` 和 `inventory.equipslots` 进行重映射。

该模组会减少或合并原版装备槽，并将部分装备的 `equippable.equipslot` 改到其他槽位。Kei 可以参考其“装备槽字符串映射”和“让装备继续进入原版 inventory 装备表”的思路，但 Kei 不减少原版装备栏，而是额外建立隐藏虚拟装备栏给头部和身体解析协议使用。

## 与真实装备共存

解析协议应允许 Kei 同时穿戴真实装备。

规则：

- 真实装备栏仍按原版逻辑工作。
- 解析协议 CD 不占用真实装备栏。
- 协议槽不视为装备栏。
- 头部和身体解析 CD 通过隐藏虚拟装备栏参与原版遍历逻辑。
- 手部解析 CD 不进入装备栏，只作为属性协议加到 Kei 身上。
- 多个解析协议同时挂载时，由协议槽数量、电量消耗和后续冲突规则限制强度。

## 协议运行成本

解析协议的运行成本按第一版实现路线区分。

```text
头部 / 身体解析协议生成的虚拟装备 -> 每 10 秒扣除 2 点电量
手部属性解析协议 -> 每 10 秒扣除 2 点数据稳定性
```

高强度装备、强模组装备或带有复杂能力的装备，可以在后续通过协议等级、耗电倍率或槽位占用规则进一步限制。

## 兼容性边界

虚拟装备路线可以提高头部和身体装备的兼容性，但仍可能遇到需要单独适配的装备。

高风险情况包括：

- 装备能力依赖专属角色组件。
- 装备能力依赖专属标签。
- 装备能力依赖专属状态图。
- 装备能力依赖专属动作或 RPC。
- 装备能力会主动移除自身、替换自身或生成绑定实体。
- 装备能力强依赖自身耐久、燃料或腐烂损耗。

这类装备即使解析为 CD，也可能需要后续建立专门适配。

攻击特效、视觉覆盖、动画覆盖等表现问题当前不作为解析协议设计边界单独讨论。

## 第一版优先级

第一版建议优先实现：

1. 解析装备生成解析协议 CD。
2. 解析协议 CD 记录 `source_prefab`、来源槽位和协议类型。
3. 为 Kei 建立隐藏虚拟装备栏，用于头部和身体解析协议。
4. 头部/身体解析 CD 插入协议槽时，生成对应虚拟装备。
5. 清理虚拟装备的拾取、装备、归属限制。
6. 将虚拟装备装备到对应隐藏虚拟装备栏。
7. 虚拟装备进入 `inventory.equipslots`，参与原版装备遍历逻辑。
8. 防御属性按原版最高值逻辑结算。
9. 虚拟装备不因耐久、燃料、腐烂等损耗被破坏。
10. 手部解析 CD 读取攻击力、位面伤害和移速倍率，作为可叠加属性加到 Kei 身上。
11. 按已挂载解析协议数量增加电量消耗。
12. 移除解析协议 CD 时关闭对应效果，并清理对应虚拟装备或属性加成。

## 暂不实现内容

第一版暂不处理：

- 自动判断强模组装备的平衡倍率。
- 为所有私有组件建立专门兼容层。
- 为依赖专属角色组件的装备补齐角色组件。
- 为依赖专属状态图或动作的装备补齐状态图逻辑。
- 将解析协议拆分为大量通用属性模块。

## 当前定案

- 可插入协议槽的协议分为战斗协议和解析协议。
- 解析协议由装备解析获得。
- 解析协议 CD 是数据载体，不视为普通装备。
- 协议槽只负责挂载 CD，不视为普通装备栏。
- 头部和身体解析 CD 生成虚拟装备，并装备到 Kei 的隐藏虚拟装备栏。
- 头部和身体虚拟装备进入 `inventory.equipslots`，尽可能继承原装备组件、标签、回调和附加能力。
- 头部和身体虚拟装备的防御属性按照原版最高值逻辑结算。
- 手部解析 CD 不生成虚拟装备，不进入 `inventory.equipslots`。
- 手部解析 CD 只读取攻击力、位面伤害和移速倍率。
- 多个手部解析 CD 可以叠加为 Kei 的属性加成。
- 解析协议不计算耐久，运行成本由协议类型承担。
- 战斗协议每 10 秒扣除 2 点电量。
- 头部 / 身体解析协议生成的虚拟装备每 10 秒扣除 2 点电量。
- 手部属性解析协议每 10 秒扣除 2 点数据稳定性。
