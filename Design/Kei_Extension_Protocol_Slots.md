# Kei 扩展路线协议槽设计

状态：当前方案已记录  
目标：定义 Kei 扩展路线的第一版核心结构，包括协议槽位、协议 CD、槽位解锁节奏和参考实现。

## 一句话定位

Kei 的扩展路线不直接装备其他角色或模组的专属物品，而是将战斗数据和装备样本转化为可挂载协议 CD，再插入自身的协议槽中挂载能力。

协议槽不是普通装备栏，而是锁定在 Kei 物品栏左侧的内部扩展单元。

生活质量 CD 与扩展路线 CD 分开处理：生活质量 CD 主要作为配方解锁材料，不进入协议槽；扩展路线 CD 才是可挂载并提供能力的物品。

## 设计来源

| 参考对象 | 用途 |
| --- | --- |
| 原版 WX-78 `wx78_inventorycontainer` | 参考“锁在物品栏格子上的不可拿起容器”。 |
| `ray` 模组额外装备栏 / 副装副武 | 参考额外容器 UI、独立于原版装备栏的槽位思路。 |
| `mygo` 模组随身听 / 唱片槽 | 参考“特定物品放入容器槽位后，角色获得对应能力”的结构。 |

当前方案主要采用 WX-78 的物品栏锁定单元结构，并参考 mygo 的“槽位内物品触发能力”逻辑。

## 核心实体

| 名称 | 暂定代码名 | 类型 | 作用 |
| --- | --- | --- | --- |
| 协议单元 | `kei_protocol_slot` | 锁定容器物品 | 占据 Kei 物品栏格子，作为协议 CD 的挂载入口。 |
| 扩展路线协议 CD | 待定 | 物品 | 可插入协议单元，激活对应战斗数据或解析协议能力。 |
| 扩展核心 | 待定 | 成长系统 / 抽象概念 | 决定 Kei 当前已解锁的协议槽数量。 |

协议单元应参考 `wx78_inventorycontainer`，但不直接复用原版 prefab。应新建 Kei 专用 prefab，避免影响 WX-78 原版逻辑。

## 物品栏结构

Kei 开局拥有 15 个物品栏格子。

其中最左侧 7 个格子预留为协议单元：

```text
[协议1][协议2][协议3][协议4][协议5][协议6][协议7][普通][普通][普通][普通][普通][普通][普通][普通]
```

设计规则：

- 开局直接生成 7 个协议单元，占据物品栏最左侧 7 格。
- 协议单元锁定在格子上，无法拿起、无法丢弃、无法被偷取。
- 协议单元不是普通背包空间。
- 最终 Kei 实际普通物品栏为 8 格。
- 牺牲普通背包空间作为扩展能力的代价。

这样可以避免后续解锁槽位时与普通物品发生位置冲突。

## 槽位解锁节奏

Kei 最多获得 7 个协议槽。

初始自带 1 个已解锁协议槽，后续每次解锁 2 个槽位，总共解锁 3 次。

| 阶段 | 解锁槽位 | 总协议槽 |
| --- | --- | --- |
| 初始 | 第 1 格 | 1 |
| Mk1 | 第 2-3 格 | 3 |
| Mk2 | 第 4-5 格 | 5 |
| Mk3 | 第 6-7 格 | 7 |

解锁顺序固定为从左到右。

```text
初始：1 可用，2-7 锁定
第一次解锁：1-3 可用，4-7 锁定
第二次解锁：1-5 可用，6-7 锁定
第三次解锁：1-7 全部可用
```

后续槽位通过右键使用槽位解锁物品解锁：

```text
Mk1 -> 解锁第 2-3 个协议槽
Mk2 -> 解锁第 4-5 个协议槽
Mk3 -> 解锁第 6-7 个协议槽
```

Mk1、Mk2、Mk3 第一版贴图均使用 `wx78module_stacksize`，临时配方均为 10 个金块。

## 锁定与解锁状态

协议单元有两种状态：

| 状态 | 表现 | 行为 |
| --- | --- | --- |
| 锁定 | 显示锁定图标或未供能状态 | 不可打开，不接受物品，不激活协议。 |
| 解锁 | 显示可用协议单元图标 | 可右键打开，可放入协议 CD。 |

实现上可参考 `wx78_inventorycontainer`：

- `inventoryitem.islockedinslot = true`：锁定在物品栏格子上。
- `inventoryitem.canbepickedup = false`：无法被正常拿起。
- `container.canbeopened = false / true`：控制锁定或解锁。
- 类似 `SetPowered` 的函数控制协议单元是否可打开。

Kei 的协议单元可以设计为：

```text
SetUnlocked(false) -> 锁定，不可打开，不接受物品
SetUnlocked(true)  -> 解锁，可打开，可插入协议 CD
```

## CD 分类规则

Kei 的 CD 分为两类，二者用途不同。

| 类型 | 用途 | 是否可放入协议槽 |
| --- | --- | --- |
| 生活质量 CD | 用作配方解锁材料，例如园艺、交易、缝补等生活协议。 | 否 |
| 扩展路线 CD | 用作可挂载协议，例如战斗协议、Boss 协议、装备解析协议、角色解析协议。 | 是 |

生活质量 CD 不进入协议槽，避免生活能力与战斗扩展能力混在同一套槽位系统里。

## 协议 CD 规则

协议槽不接受普通物品，也不直接接受普通装备。

第一版只允许插入带有扩展路线协议标签的 CD：

```text
kei_protocol_cd
kei_extension_cd
```

协议槽可接受的 CD 分为：

| 类型 | 来源 | 作用 |
| --- | --- | --- |
| 战斗数据 CD | 数据记录器完成巨兽记录后获得 | 提供 Boss 战斗数据协议和对应战斗 buff。 |
| 解析协议 CD | 解析外部装备生成 | 头/身生成虚拟装备，手部提供属性加成。 |
| 异常协议 CD | 暗影、月亮、远古数据生成 | 提供特殊环境和异常力量适配。 |
| 角色协议 CD | 后续暂定 | 可能用于角色机制学习，第一版暂不展开。 |

协议槽本身只负责挂载，协议 CD 决定具体能力。

不允许放入协议槽的 CD：

- 生活质量 CD。
- 教材类 CD。
- 未完成的纯数据样本 CD。
- 空白 CD。
- 已绑定但未完成记录的战斗数据 CD。
- 任意没有扩展路线协议标签的普通唱片或数据记录物品。

## 协议挂载流程

协议挂载的基本流程：

```text
获得战斗数据 CD / 解析协议 CD
-> 打开已解锁协议单元
-> 放入协议 CD
-> Kei 激活对应协议能力
```

移除协议 CD 时：

```text
从协议单元取出协议 CD
-> Kei 关闭对应协议能力
```

参考 mygo 随身听：

- `itemget` 时读取槽内物品并激活能力。
- `itemlose` 时移除对应能力。
- 物品更换持有者或离开角色时，关闭能力。

## 战斗数据与协议槽的关系

完成记录后的战斗数据 CD 是可挂载的战斗协议 CD。

当前流程：

```text
数据记录器完成记录 -> 战斗数据 CD -> 插入协议槽 -> 战斗协议组件读取并生效
```

示例：

| 战斗数据 CD | 挂载效果 |
| --- | --- |
| 独眼巨鹿战斗数据 CD | 攻击附带冻结，免疫过热状态和过热扣血。 |
| 熊獾战斗数据 CD | 攻击造成群体伤害。 |
| 龙蝇战斗数据 CD | 攻击点燃目标，对燃烧目标增伤，免疫寒冷状态和寒冷扣血。 |

这样战斗数据的用途由协议槽承接：

- 玩家可以保存多种 Boss 战斗数据。
- 玩家需要选择当前挂载哪几个战斗数据协议。
- 槽位限制决定扩展路线强度上限。

生活数据不走这个流程。生活数据产生的教材或生活质量 CD 主要作为配方解锁材料，用于解锁生活能力，不作为协议槽挂载物。

## 装备解析与适配

Kei 不直接装备其他角色的专属装备，也不强制绕过原作者的装备限制。

当前流程：

```text
解析装备
-> 生成解析协议 CD
-> 插入协议槽
-> 根据装备部位进入不同生效路线
```

解析协议分为两类：

| 解析类型 | 生效方式 |
| --- | --- |
| 头部 / 身体解析 CD | 插槽后后台生成虚拟装备，装备到 Kei 的隐藏虚拟装备栏，并进入 `inventory.equipslots`。 |
| 手部解析 CD | 不生成虚拟装备，只读取攻击力、位面伤害和移速倍率，作为属性协议加到 Kei 身上。 |

头部和身体解析协议的目标是尽可能继承原装备组件、标签、回调和附加能力。防御属性进入原版 `inventory:ApplyDamage()` 结算，多件护甲吸收率按原版最高值逻辑处理。

手部解析协议更接近战斗协议的属性 buff。多个手部解析 CD 可以叠加，但不继承武器主动技能、远程投射物、攻击特效、视觉覆盖或必须真实手持才能触发的逻辑。

保留兼容性边界：

- 很多模组装备效果写在私有函数中。
- 部分能力依赖角色组件、状态图或专属标签。
- 自动完整继承仍可能破坏兼容性或造成崩溃。

## 协议运行成本

协议挂载不应完全免费。

可选成本：

- 持续消耗电量。
- 持续消耗数据稳定性。
- 高级协议占用更多槽位。
- 协议之间存在冲突标签。
- 过载状态下临时启用额外协议，但结束后产生副作用。

第一版可以先只做槽位限制，后续再增加能量和稳定性消耗。

## 槽位保护与自检

协议单元是 Kei 扩展路线的核心系统，必须具备较强的保护和兼容能力。

每次进入游戏、角色生成、读档完成或角色物品栏初始化后，都应检查 7 个协议单元是否正常存在。

自检目标：

```text
确认最左侧 7 个物品栏格子存在协议单元
确认协议单元顺序正确
确认协议单元数量为 7
确认每个协议单元锁定在对应格子
确认槽位解锁进度与角色保存数据一致
确认未解锁槽位不可打开
确认已解锁槽位可打开并可接受扩展路线协议 CD
```

如果检测到协议单元缺失：

- 在对应格子补生成协议单元。
- 新生成的协议单元根据当前解锁进度设置锁定 / 解锁状态。
- 不应额外增加超过 7 个协议单元。

如果检测到协议单元位置错误：

- 优先尝试移动回最左侧对应格子。
- 如果该格子被普通物品占用，优先将普通物品移动到其他可用物品栏位置。
- 如果没有空位，普通物品可以掉落到玩家脚下，协议单元回到固定格子。

如果检测到多余协议单元：

- 移除多余协议单元。
- 如果多余协议单元内含有协议 CD，优先将 CD 返还到玩家物品栏；没有空位时掉落到玩家脚下。

如果检测到未解锁槽位中存在 CD：

- 关闭该槽位。
- 取出 CD 并返还给玩家；没有空位时掉落到玩家脚下。
- 重新应用该槽位的锁定状态。

## 解锁进度保存

槽位解锁进度不应只依赖协议单元本身，而应保存在 Kei 的角色数据或扩展核心组件中。

建议保存字段：

```text
protocol_slot_unlock_level = 0 / 1 / 2 / 3
```

对应关系：

| 解锁等级 | 已解锁槽位 |
| --- | --- |
| 0 | 第 1 格 |
| 1 | 第 1-3 格 |
| 2 | 第 1-5 格 |
| 3 | 第 1-7 格 |

读档或进游戏时，根据 `protocol_slot_unlock_level` 重新设置 7 个协议单元的状态。

## 设计边界

- 协议槽不是普通装备栏。
- 协议槽不直接放普通装备。
- 协议槽只接受战斗相关 CD 和解析类扩展路线 CD。
- 生活质量 CD 只作为配方解锁材料，不放入协议槽。
- 协议单元锁定在物品栏左侧，不能被拿起或丢弃。
- 未解锁协议单元不可打开，也不接受物品。
- 协议 CD 插入后才激活能力，移除后能力关闭。
- 完成记录后的战斗数据 CD 可以直接作为战斗协议挂载。
- 头部和身体解析协议尽量继承原装备效果；手部解析协议只读取攻击力、位面伤害和移速倍率。
- 每次进入游戏时必须检查并修复 7 个协议单元和槽位解锁状态。
