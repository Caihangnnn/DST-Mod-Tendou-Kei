# Kei 战斗数据记录器设计

状态：当前方案已固定  
目标：定义 Kei 学习路线中“战斗数据”的第一版获取方式，包括数据记录 CD、数据记录器、记录领域、交互流程、状态机和异常处理。

## 一句话定位

战斗数据记录器是 Kei 用于学习巨兽战斗样本的实验设备。玩家需要先部署记录领域，再用数据记录 CD 绑定领域内的巨兽，最后通过击败该巨兽完成战斗数据记录。

这套系统用于回答两个问题：

- Kei 如何从世界学习战斗规则？
- Boss / 巨兽数据如何转化为能力解锁材料？

## 参考对象

| 参考对象 | 用途 |
| --- | --- |
| 原版 `record` | 临时作为数据记录 CD 的贴图来源。 |
| 原版 `wagpunk_cagewall` | 作为记录领域边缘视觉效果参考。 |
| 原版 `wagpunk_arena_collision` | 作为竞技场边界 / 力场规则参考。 |
| 星野模组“卢尼火”类实现 | 参考“普通伤害不可摧毁，但玩家可用锤子拆除”的结构。 |

需要注意：原版 `wagpunk_cagewall` 主要承担视觉表现，真正的竞技场阻挡逻辑由碰撞和场地规则完成。Kei 的数据记录领域也应拆成“视觉层”和“规则层”，避免把视觉墙和逻辑墙写死在一起。

## 核心物品与实体

| 名称 | 暂定代码名 | 类型 | 作用 |
| --- | --- | --- | --- |
| 数据记录 CD | 待定，临时简称 `CD` | 物品 | 空白数据介质，作为绑定巨兽和记录战斗数据的载体。 |
| 已绑定数据记录 CD | 待定 | 物品状态 | 记录了目标巨兽信息，但尚未完成战斗数据记录。 |
| 战斗数据 CD | 待定 | 物品状态 / 产物 | 记录完成后的 CD，可作为解锁战斗能力、Boss 协议或协议槽升级的材料。 |
| 数据记录器 | 待定 | 可部署建筑 | 管理绑定、记录、力场开启 / 关闭、完成和收获流程。 |
| 记录领域 | 待定 | 领域逻辑 + 视觉效果 | 记录期间开启力场边界，阻止生物进出。 |

第一版可以先共用原版 `record` 的贴图，让空白 CD、已绑定 CD、战斗数据 CD 通过名称、描述和内部数据区分。

## 部署规则

数据记录器是实验设备，不作为障碍物或建筑防线使用。

- 数据记录器没有碰撞体积。
- 数据记录器可以在任意地点部署。
- 数据记录器部署时不检查地形以外的复杂建造条件。
- 数据记录器不阻挡玩家、生物、物品或投射物。
- 数据记录器的领域效果只在记录中状态开启。
- 数据记录器的部署位置决定记录领域中心。

## 记录领域规则

数据记录器部署后，只显示设备本体，不立即开启完整力场。玩家将已绑定数据记录 CD 给予数据记录器后，数据记录器才会在自身周围开启固定范围的记录领域。

- 空闲状态下，记录领域关闭。
- 记录中状态下，记录领域开启。
- 领域边缘显示类似 `wagpunk_cagewall` / 瓦格斯塔夫竞技场力场的蓝绿色蜂窝状视觉效果。
- 领域规则优先参考原版瓦格斯塔夫竞技场力场。
- 领域开启后，阻止生物从内部离开，也阻止外部生物进入。
- 绑定的巨兽被限制在领域内，直到记录完成或玩家主动停止记录。
- 领域第一版可以暂不限制物品、投射物和玩家召唤物。

实现上优先借鉴原版力场 / 碰撞规则，而不是第一版自写软边界：

```text
提交已绑定 CD -> 开启力场 -> 阻止生物进出 -> 记录完成 / 主动停止 -> 关闭力场
```

如果原版力场会影响玩家进出，设计上接受这种竞技场式封闭体验，但必须提供可靠的主动停止手段，避免玩家打不过 Boss 时被困死。

## 交互流程

固定交互流程如下：

```text
1. 玩家制作空白“数据记录 CD”。
2. 玩家部署“数据记录器”。
3. 数据记录器处于空闲状态，力场关闭。
4. 玩家手持空白 CD，将鼠标放在记录器范围内拥有 epic 标签的巨兽实体上。
5. 玩家左键点击巨兽，将 CD 与该巨兽绑定。
6. CD 进入“已绑定”状态，记录目标 prefab / GUID / 显示名等信息。
7. 玩家将已绑定 CD 给予数据记录器。
8. 数据记录器检查 CD 绑定的巨兽是否仍在记录领域范围内。
9. 若目标不在范围内，提交失败，CD 不被消耗，记录器保持空闲。
10. 若目标在范围内，数据记录器进入“记录中”状态，并开启力场。
11. 玩家在记录领域内击败被绑定的巨兽。
12. 若绑定巨兽死亡时仍满足记录条件，数据记录器进入“完成”状态。
13. 数据记录器关闭力场。
14. 玩家左键点击数据记录器，收获“战斗数据 CD”。
15. 数据记录器回到初始空闲状态，可进行下一次记录。
```

这套流程中，CD 负责保存样本目标，数据记录器负责保存记录状态。这样比“CD 绑定后自动记录”更稳定，也方便处理保存、失败和重置。

## 主动停止记录

记录过程中必须提供保护方案，避免玩家打不过 Boss 时被力场困住。

固定方案：

```text
记录中 -> 玩家右键点击数据记录器 -> 停止记录 -> 关闭力场 -> 返还空白 CD -> 回到空闲
```

- 主动停止只在“记录中”状态可用。
- 主动停止不会产出战斗数据。
- 主动停止会清除 CD 上的绑定信息，返还空白数据记录 CD。
- 主动停止会立即关闭力场。
- 主动停止后，数据记录器回到空闲状态。
- 右键停止应优先使用清晰的动作名，例如“停止记录”或“中止记录”。

## 绑定规则

手持空白 CD 点击目标时，需要满足以下条件：

- 目标必须拥有 `epic` 标签。
- 目标必须在某个数据记录器的有效记录范围内。
- 目标必须是有效实体，且不能已经死亡。
- 目标不能处于 `INLIMBO`、不可见或无法记录的特殊状态。
- 空白 CD 没有绑定过其他目标。
- 同一目标可以被不同数据记录器绑定与否，后续根据实现复杂度决定；第一版建议同一时间只允许一个记录器记录同一目标。

绑定成功后，CD 记录：

```text
data_type = "combat"
target_prefab = 目标 prefab
target_guid = 目标 GUID
target_name = 目标显示名
bound = true
completed = false
```

如果目标是模组 Boss，优先记录 prefab。后续可通过 prefab 建立对应的 Boss 协议或通用 epic 战斗数据。

## 提交规则

玩家将已绑定数据记录 CD 给予数据记录器时，需要再次校验目标状态。

提交成功条件：

- 数据记录器处于空闲状态。
- CD 为已绑定状态。
- CD 绑定的目标仍然有效。
- CD 绑定的目标仍拥有 `epic` 标签。
- CD 绑定的目标没有死亡。
- CD 绑定的目标没有处于 `INLIMBO`。
- CD 绑定的目标位于该数据记录器的记录领域范围内。

如果目标不在该记录器的记录领域范围内，则提交失败：

- 数据记录器不进入记录中状态。
- 力场不开启。
- CD 不被消耗。
- CD 仍保持已绑定状态。
- 玩家需要把目标引入记录范围，或换一台更合适位置的数据记录器。

这条规则用于避免玩家先绑定远处 Boss，再把 CD 交给任意位置的记录器远程开启记录。

## 数据记录器状态机

数据记录器固定为四个主要状态。

| 状态 | 说明 | 可交互行为 |
| --- | --- | --- |
| 空闲 | 已部署，力场关闭，没有记录任务。 | 可接收已绑定 CD。 |
| 记录中 | 已接收绑定 CD，力场开启，正在追踪目标巨兽。 | 不可接收新 CD；可右键停止记录。 |
| 完成 | 目标死亡且记录条件满足，数据记录完成，力场关闭。 | 玩家左键点击收获战斗数据 CD。 |
| 异常 | 目标、CD 或记录条件失效，力场关闭。 | 返还 / 损坏 CD，或清除记录后回到空闲。 |

### 空闲状态

- 数据记录器刚部署后进入空闲状态。
- 力场关闭。
- 可接受已绑定数据记录 CD。
- 接收已绑定 CD 时，必须确认绑定目标仍在记录范围内。
- 不接受空白 CD。
- 不接受已完成的战斗数据 CD。

### 记录中状态

- 数据记录器保存 CD 里的目标信息。
- 数据记录器开启力场。
- 数据记录器追踪目标实体。
- 记录中状态下领域视觉可以增强，例如提高边缘闪烁频率或播放记录音效。
- 如果目标死亡并满足条件，进入完成状态。
- 如果目标失效，进入异常状态。
- 玩家可以右键点击数据记录器，主动停止记录。

### 完成状态

- 数据记录器内部保存完成结果。
- 力场关闭。
- 玩家左键点击数据记录器后，产出战斗数据 CD。
- 收获完成后，数据记录器清空当前记录并回到空闲状态。

### 异常状态

异常状态用于避免设备卡死。第一版可以不做复杂 UI，只需要明确处理结果。

进入异常状态后，优先尝试返还绑定 CD；如果 CD 或目标信息已不可恢复，则清除记录并回到空闲。

### 主动停止状态处理

主动停止不是异常，而是玩家主动放弃本次记录。

- 关闭力场。
- 清除当前目标绑定。
- 返还空白数据记录 CD。
- 不产出战斗数据。
- 数据记录器回到空闲状态。

## 异常处理

以下情况应视为异常：

| 异常情况 | 处理方式 |
| --- | --- |
| 绑定目标离开记录领域 | 记录中断，进入异常状态。可返还已绑定 CD。 |
| 绑定目标被删除或 `Remove` | 记录中断，进入异常状态。 |
| 绑定目标进入 `INLIMBO` | 暂停追踪或进入异常状态；第一版建议直接异常。 |
| 绑定目标死亡时不在领域内 | 记录失败，进入异常状态。 |
| 提交 CD 时绑定目标不在记录范围内 | 提交失败，CD 不消耗，记录器保持空闲。 |
| 数据记录器被玩家锤掉 | 中断记录，返还或掉落内部 CD，移除领域。 |
| 世界保存 / 读取后目标 GUID 失效 | 进入异常状态，避免卡死。 |
| 玩家给予的 CD 状态不合法 | 拒绝给予，不改变记录器状态。 |
| 数据记录器已有记录任务 | 拒绝新的 CD。 |

第一版建议采用保守规则：只要目标不再能被稳定追踪，就中断记录并回到空闲。后续如果需要降低挫败感，可以再加入短时间容错。

玩家主动右键停止记录不视为异常，按“主动停止记录”流程处理。

## 摧毁与拆除规则

数据记录器不应被普通战斗破坏。

- 不通过普通 `combat` 或 `health` 方式被生物摧毁。
- 不被巨兽踩踏、攻击或范围伤害破坏。
- 玩家可以使用锤子拆除。
- 被锤子拆除时，移除记录领域和视觉效果。
- 若正在记录，优先掉落或返还内部 CD。
- 若正在记录，锤子拆除会关闭力场。

这部分实现上可以参考“只保留 `workable` 的 `HAMMER` 动作，不提供普通生命值破坏入口”的写法。

## 战斗数据 CD 用途

战斗数据 CD 是学习路线中战斗能力的解锁材料之一。

第一版用途可以分为三层：

| 用途 | 示例 |
| --- | --- |
| 通用战斗数据 | 解锁基础战斗协议、弱点分析、轨道炮基础强化。 |
| Boss 级战斗数据 | 解锁高级战斗模块、战斗协议槽位、Boss 特攻。 |
| 指定 Boss 数据 | 解锁对应 Boss 协议，例如冰霜抗性、冲撞预测、暗影战斗协议、月亮战斗协议。 |

战斗数据 CD 不直接等于能力。它更像是“研究材料”，需要与数据终端、配方或模块升级系统结合，才能转化为 Kei 的能力。

## 设计边界

- 数据记录器记录的是战斗过程和死亡样本，不是直接复制 Boss 能力。
- 只有 `epic` 目标参与第一版战斗数据记录。
- 记录领域服务于记录流程，不设计成通用困怪神器。
- 数据记录器没有碰撞体积，可以在任意地点部署。
- 绑定目标必须在记录范围内才能提交 CD 并开始记录。
- 力场只在记录中开启，空闲、完成、异常和主动停止后均关闭。
- 若力场会造成玩家进出限制，必须保留右键停止记录作为保护方案。
- 第一版优先使用原版力场规则，减少自定义边界逻辑。
- 第一版只处理战斗数据，其他数据类型后续单独设计。

## 后续待细化

1. 确定数据记录 CD、战斗数据 CD、数据记录器的代码名。
2. 确定记录领域半径。
3. 确定 CD 绑定动作是复用 `GIVE`、自定义动作，还是类似 `recipescanner` 的物品对目标动作。
4. 确定数据记录器的部署配方。
5. 确定记录完成后产出的战斗数据 CD 是否区分 Boss prefab。
6. 确定哪些 Boss 数据对应哪些战斗能力或协议解锁。
