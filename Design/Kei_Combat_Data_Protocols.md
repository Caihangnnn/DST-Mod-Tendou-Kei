# Kei 战斗数据协议设计

本文档记录 Kei 通过数据记录器获取巨兽战斗数据后，将战斗数据 CD 插入协议槽并获得增益的当前方案。

## 1. 定位

战斗数据 CD 属于“扩展路线 CD”，用于插入 Kei 的协议槽并提供战斗增益。

它与解析协议不同：

- 战斗数据 CD 不进入虚拟装备栏。
- 战斗数据 CD 不生成虚拟装备。
- 战斗数据 CD 不继承装备本体的 `equippable`、护甲、防具或武器逻辑。
- 战斗数据 CD 通过 Kei 身上的协议管理组件或容器逻辑，为角色挂载增益 buff。

协议槽可以同时支持多种扩展 CD，但进入槽位后需要先判断 CD 类型：

- `combat`：战斗数据协议，进入战斗数据增益系统。
- `analysis`：解析协议，进入解析协议和虚拟装备系统。
- `life`：生活质量 CD 不允许插入协议槽，主要作为配方解锁或制作材料使用。

## 2. 参考实现方向

当前战斗协议的实现最接近 `Reference/mygo/scripts/prefabs/mygo_cd_player.lua`：

- 容器监听物品放入和移除。
- 每次槽位变化时重新扫描所有槽位。
- 根据 CD 记录的数据名收集当前激活的数据。
- 相同数据只保留一份有效结果。
- 对新出现的数据执行启用逻辑。
- 对消失的数据执行移除逻辑。

可参考的逻辑结构：

```lua
local active = {}

for slot, item in pairs(container.slots) do
    if item ~= nil and item:HasTag("kei_combat_protocol_cd") then
        local data_id = item.components.kei_protocoldata ~= nil
            and item.components.kei_protocoldata:GetDataID()
            or nil

        if data_id ~= nil and active[data_id] == nil then
            active[data_id] = item
        end
    end
end

owner.components.kei_combatprotocol:Rebuild(active)
```

实际编码时可以命名为：

- `kei_protocolhost`：负责协议槽整体扫描、分发和安全检查。
- `kei_combatprotocol`：只负责战斗协议的启用、刷新和移除。
- `kei_protocoldata`：挂在 CD 上，记录该 CD 的数据类型、来源 prefab、协议 id。

## 3. CD 数据结构

战斗数据 CD 需要记录来源巨兽和协议类型。

建议字段：

```lua
{
    protocol_type = "combat",
    combat_data_id = "deerclops",
    boss_prefab = "deerclops",
}
```

建议标签：

- `kei_protocol_cd`
- `kei_combat_protocol_cd`
- `kei_boss_data_cd`

## 4. 生效规则

### 4.1 相同战斗数据不叠加

同一种战斗数据只生效一次。

例如，协议槽中同时插入两个独眼巨鹿战斗数据 CD 时：

- 两个 CD 都可以保留在槽位中。
- 只有一份独眼巨鹿协议效果生效。
- 不增加冻结概率、冻结强度或免疫效果强度。

是否在 UI 上提示“重复协议未额外生效”，可以留到交互实现阶段再决定。

### 4.2 与解析协议的关系

战斗协议只修改 Kei 自身的战斗 buff 或角色状态，不参与解析协议的虚拟装备栏。

解析协议中的身体、头部装备仍然通过虚拟装备栏继承装备效果；战斗协议则通过组件、标签、事件监听、属性修饰器等方式直接挂到角色身上。

### 4.3 资源代价

挂载战斗协议越多，数据稳定性下降越快。

该规则与解析协议的电量消耗区分：

- 解析协议越多，电量消耗越快。
- 战斗协议越多，数据稳定性下降越快。

## 5. 初始可记录巨兽

第一版暂定只开放三个可记录巨兽：

- 独眼巨鹿：`deerclops`
- 熊獾：`bearger`
- 龙蝇：`dragonfly`

数据记录器完成记录后，根据绑定目标 prefab 生成对应战斗数据 CD。

当前阶段建议采用白名单：

- 绑定时仍然要求目标拥有 `epic` 标签。
- 提交 CD 时仍然要求绑定目标在数据记录器领域内。
- 若目标 prefab 不在战斗数据白名单中，则不生成可挂载战斗协议 CD。

后续可以扩展为“未知巨兽数据”或“未解析战斗数据”，但第一版先保持明确可控。

## 6. 独眼巨鹿战斗数据

来源 prefab：`deerclops`

协议 id：`deerclops`

建议显示名：独眼巨鹿战斗数据

### 6.1 效果

插入协议槽并生效后：

- Kei 的攻击附带冻结效果。
- Kei 免疫过热状态。
- Kei 免疫过热造成的生命值损失。

### 6.2 实现方向

冻结效果可以通过监听 Kei 的攻击事件实现：

- 监听 `onattackother` 或等价的攻击命中事件。
- 若目标拥有 `freezable` 组件，则对目标增加冰冻值。
- 冻结值、触发概率、内置冷却可以后续根据强度调整。

过热免疫直接参考 functional medal 中温度免疫类装备的实现方式：

- 给 Kei 添加协议来源的过热免疫标记。
- 在协议生效期间阻止温度超过过热阈值，或阻止过热扣血逻辑。
- 协议移除时撤销对应标记和修饰。

这里不重新设计温度系统，按 functional medal 的处理思路同时覆盖“过热状态”和“过热扣血”。

## 7. 熊獾战斗数据

来源 prefab：`bearger`

协议 id：`bearger`

建议显示名：熊獾战斗数据

### 7.1 效果

插入协议槽并生效后：

- Kei 的攻击造成群体伤害。

### 7.2 实现方向

群体伤害可以参考功能勋章中蜂王勋章的范围攻击处理方式：

- Kei 攻击命中目标后，以目标或 Kei 为中心搜索附近实体。
- 筛选可被攻击、非友方、未死亡、非玩家保护对象的目标。
- 对筛选出的目标造成一次额外伤害。
- 使用内部标记避免范围伤害再次触发范围伤害，导致递归。

伤害参数暂不固定，后续可在平衡阶段决定：

- 按本次攻击伤害的一定比例造成。
- 或使用固定基础值加 Kei 属性修正。
- 或对主目标不重复结算，只影响周围目标。

## 8. 龙蝇战斗数据

来源 prefab：`dragonfly`

协议 id：`dragonfly`

建议显示名：龙蝇战斗数据

### 8.1 效果

插入协议槽并生效后：

- Kei 的攻击附带点燃效果。
- Kei 对燃烧目标造成的伤害增加。
- Kei 免疫寒冷状态。
- Kei 免疫寒冷造成的生命值损失。

### 8.2 实现方向

点燃效果可以参考原版 Willow 的火焰相关能力：

- 监听 Kei 的攻击命中事件。
- 若目标拥有 `burnable` 组件且允许燃烧，则调用点燃逻辑。
- 对不应被点燃的目标遵循原版组件判断，不强行绕过燃烧免疫。

对燃烧目标增伤可以在攻击结算前后处理：

- 若目标正在燃烧，则为本次攻击增加伤害倍率。
- 或在命中后追加一段额外火焰伤害。

寒冷免疫直接参考 functional medal 中防寒类装备的实现方式：

- 给 Kei 添加协议来源的寒冷免疫标记。
- 在协议生效期间阻止温度低于寒冷阈值，或阻止寒冷扣血逻辑。
- 协议移除时撤销对应标记和修饰。

这里不重新设计温度系统，按 functional medal 的处理思路同时覆盖“寒冷状态”和“寒冷扣血”。

## 9. 启用和移除

战斗协议推荐统一走组件接口，而不是把逻辑分散写在 CD 或槽位物品上。

建议接口：

```lua
function KeiCombatProtocol:ApplyProtocol(data_id, source_item)
end

function KeiCombatProtocol:RemoveProtocol(data_id)
end

function KeiCombatProtocol:Rebuild(active_protocols)
end
```

`Rebuild` 的职责：

- 读取当前槽位中所有战斗协议 CD。
- 对重复协议去重。
- 找出新增协议并启用。
- 找出消失协议并移除。
- 保留仍然存在的协议，不重复添加监听器或修饰器。

所有事件监听、标签、属性修饰器都必须能被干净移除，避免玩家切换 CD 后残留效果。

## 10. 当前未定参数

以下数值暂不固定，留到实现和测试阶段决定：

- 独眼巨鹿协议的冻结触发概率。
- 独眼巨鹿协议每次攻击增加的冰冻值。
- 熊獾协议的范围伤害半径。
- 熊獾协议的范围伤害倍率。
- 龙蝇协议的点燃概率。
- 龙蝇协议对燃烧目标的增伤倍率。
- 战斗协议数量对数据稳定性下降速度的影响。
